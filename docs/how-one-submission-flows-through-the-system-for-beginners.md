# 给代码小白的一次提交故事版

这份文档不讲很多术语。

它只讲一件事：

**当一个用户提交一段代码后，系统里到底发生了什么。**

你可以把它当成一段“故事”。

故事的主角不是人，而是一份代码提交。

---

## 一、先假装你是用户

你现在写了一段代码，比如：

```cpp
#include <iostream>
int main() {
    int a, b;
    std::cin >> a >> b;
    std::cout << a + b << std::endl;
    return 0;
}
```

你把这段代码发给评测系统。

你心里想的是：

“帮我看看这题写对了没有。”

系统收到之后，真正要做的事情其实很多。

它不是直接一句“对/错”就结束了。

它要经过很多步骤。

---

## 二、第一站：网络入口收到你的请求

先到这里：

- `main.cpp`
- `src/TcpServer.cpp`
- `src/client_sockets.cpp`

你可以把这一层理解成“前台接待”。

它做的事是：

1. 打开一个端口，等待客户端连接
2. 收到一条消息
3. 先把消息完整收下来
4. 再交给后面的模块处理

这里最重要的一点是：

**系统先关心“有没有收到完整消息”，还不关心你的代码对不对。**

换句话说，这一层处理的是“通信问题”，不是“判题问题”。

---

## 三、第二站：协议层把消息翻译成人话

接下来会到：

- `include/protocol/JudgeProtocol.h`
- `src/protocol/JudgeProtocol.cpp`

这一层就像翻译官。

客户端发来的原始内容，本来是 JSON 文本，比如类似这样：

```json
{
  "uuid": 123,
  "pid": "1000",
  "lang": 0,
  "code": "...用户代码..."
}
```

协议层的工作就是把它翻译成程序内部能直接使用的结构：

- `SubmissionRequest`

翻译之后，系统终于知道了这 4 件事：

1. 这是哪次请求（`uuid`）
2. 这是哪道题（`pid`）
3. 这是什么语言（`lang`）
4. 用户交了什么代码（`code`）

所以你可以把协议层理解成：

**把“网络消息”变成“程序任务单”。**

---

## 四、第三站：服务层接过这张任务单

真正的大脑在这里：

- `include/service/SubmissionService.h`
- `src/service/SubmissionService.cpp`

你可以把 `SubmissionService` 理解成“总调度员”。

它会拿着 `SubmissionRequest` 想：

“好，现在我该一步一步安排这次评测了。”

它做的第一件事，不是直接编译。

而是：

### 1. 给这次提交分一个内部编号

这个编号叫：

- `submission_id`

这是服务端自己发的编号。

为什么要自己发？

因为客户端带来的 `uuid` 不一定安全、不一定唯一。

服务端自己发一个，更适合内部管理。

---

## 五、第四站：系统先记录“我正在处理这份提交”

这时 `SubmissionService` 会把当前状态写进：

- `ResultStore`

状态会变成：

- `PREPARING`

这表示：

**“我已经收到提交了，现在开始准备。”**

你可以把 `ResultStore` 理解成“进度记录本”。

它不是去跑代码的人。

它只是记着：

- 现在做到哪一步了
- 当前结果是什么
- 每个测试点结果是什么

---

## 六、第五站：系统决定该找谁来执行这段代码

接下来会到：

- `src/runner/RunnerFactory.cpp`

这一步很像“分配老师”。

系统看一眼语言字段：

- 如果是 `C++`，就交给 `CppRunner`
- 如果是 `Python`，就交给 `PythonRunner`

也就是说：

**总流程差不多，但不同语言找不同的人处理。**

这就是为什么项目后面还能扩展更多语言。

以后如果你再加一种语言，比如 `Java`，理论上就是再加一个新的 runner。

---

## 七、第六站：runner 先做准备工作

runner 拿到任务后，先做：

- `prepare()`

这一步是什么意思？

就是先把“环境准备好”。

比如：

- 创建工作目录
- 把代码写入文件

如果是 `C++`，它可能会写成：

- `solution.cpp`

如果是 `Python`，它可能会写成：

- `solution.py`

所以 `prepare()` 不是在跑代码。

它是在说：

**“先把试卷和考场准备好。”**

---

## 八、第七站：进入编译或语法检查阶段

这时状态会更新成：

- `COMPILING`

然后 runner 会做：

- `compile()`

### 如果是 C++

会真的调用编译器，比如：

- `g++`

### 如果是 Python

虽然 Python 不是传统编译型语言，但这里也会统一走一个“检查阶段”，比如：

- `python3 -m py_compile`

这样设计的好处是：

不管哪种语言，服务层都可以用同一种思路处理：

- 先准备
- 再检查/编译
- 再运行

这会让整个系统更整齐。

---

## 九、第八站：如果编译失败，会发生什么

如果代码在这一步出问题，比如：

- 少分号
- 语法错误

那么系统不会再继续跑测试点了。

它会直接把结果写成：

- `status = FINISHED`
- `verdict = CE`

意思是：

**“这次流程已经结束了，结论就是编译错误。”**

注意这里很重要：

`FINISHED` 不等于 `AC`。

它只是表示：

**流程结束了。**

---

## 十、第九站：如果编译成功，就开始逐个测试点运行

如果编译/检查没问题，系统会把状态改成：

- `RUNNING`

然后它会根据题目编号 `pid` 去找测试数据。

它会去看类似这样的目录：

- `testData/1000/data`

这里面会有很多输入输出文件。

比如：

- `problem1.in`
- `problem1.out`

这就像老师拿出标准测试样例，准备开始一道一道验。

---

## 十一、第十站：runner 开始跑每一个测试点

这一层会用：

- `runCase()`

它拿到的信息包括：

- 这是第几个测试点
- 输入文件在哪里
- 正确输出在哪里
- 时间限制
- 内存限制

然后它真的去执行程序。

如果是 `C++`，执行的是编译出来的可执行文件。

如果是 `Python`，执行的是 Python 脚本。

跑完之后，会得到一个单点结果：

- `SubmissionCaseResult`

这里会记录：

- 这个点过没过
- 用了多少时间
- 占了多少内存
- 是否异常退出

---

## 十二、第十一步：系统检查输出对不对

程序跑完，不代表就通过了。

系统还要比较：

- 用户程序输出了什么
- 标准答案是什么

如果部署了 checker，就优先用 checker。

如果没有，就走系统内部的 fallback 比较逻辑。

这样系统才知道这个测试点到底是：

- `AC`
- `WA`
- `RE`
- `TLE`

所以单个测试点不是只看“程序有没有跑完”，还要看“答案对不对”。

---

## 十三、第十二站：每跑完一个点，系统就更新一次记录

这一步很容易被忽视，但很重要。

系统不是等所有测试点都跑完了才记结果。

它是每跑一个点，就把这个点的结果写到：

- `ResultStore`

这意味着 `ResultStore` 里保存的是一张“动态更新的成绩单”。

你可以理解成：

**老师每改完一道小题，就先记一笔。**

---

## 十四、第十三站：所有测试点跑完后，系统开始做总判定

等所有点都跑完，系统会把这些单点结果交给：

- `JudgeCore`

这一层的工作不是执行代码，而是：

**把多个测试点结果合并成一个总结果。**

比如：

- 全部 `AC` -> 总结果 `AC`
- 有 `CE` -> 总结果 `CE`
- 有超时 -> 总结果可能是 `TLE`

这一步很像老师改完整张卷子后，最后在最上面写一个总评。

---

## 十五、第十四站：系统把最终结果保存下来

这时系统会生成一个总结果快照：

- `SubmissionResult`

里面会包含：

- `submission_id`
- 当前状态
- 最终 verdict
- 错误/提示信息
- 所有测试点结果

然后写回 `ResultStore`。

状态一般会变成：

- `FINISHED`

到这里，这次评测流程就算真正结束了。

---

## 十六、第十五站：协议层把结果翻译回 JSON

前面协议层做的是：

- JSON -> `SubmissionRequest`

现在它做的是反方向：

- `SubmissionResult` -> JSON

也就是：

系统内部能懂的结果，要重新翻译成客户端能懂的格式。

最后会带上这些信息：

- `submission_id`
- `status`
- `verdict`
- `message`
- `case_results`

---

## 十七、第十六站：网络层把结果回给客户端

最后又回到：

- `src/client_sockets.cpp`

这一层会把结果：

1. 加上长度前缀
2. 放进待发送缓冲区
3. 在 socket 可写时发回给客户端

于是客户端最终收到了一份结果。

这就是你最开始想得到的那句答案：

- 通过
- 错误
- 超时
- 编译失败

---

## 十八、如果中途失败，会在哪些地方失败

你可以把失败分成几类：

### 1. 请求本身就有问题

发生在：

- `JudgeProtocol`

比如：

- JSON 格式不对
- 缺少字段
- 语言字段不合法

### 2. 系统不支持这种语言

发生在：

- `RunnerFactory`

比如：

- 传进来的是系统当前不支持的语言

### 3. 准备阶段失败

发生在：

- runner 的 `prepare()`

比如：

- 工作目录创建失败
- 文件写入失败

### 4. 编译阶段失败

发生在：

- runner 的 `compile()`

比如：

- `C++` 语法错误
- `Python` 语法检查不通过

### 5. 运行阶段失败

发生在：

- runner 的 `runCase()`

比如：

- 程序崩溃
- 非零退出
- 超时

### 6. 系统级失败

发生在：

- `SubmissionService`
- `ResultStore`
- 内部异常处理

比如：

- 找不到测试数据
- 内部抛异常

---

## 十九、把整个故事压缩成一句话

如果你只能记一句，那就记这句：

**客户端发来一段代码，系统先把它翻译成内部任务，再按语言选择执行器，跑完所有测试点，汇总结果，保存结果，最后再把结果翻译回 JSON 发回去。**

---

## 二十、你现在应该能回答的 5 个问题

如果这份故事你读懂了，你应该已经能回答：

1. 用户代码最先进入哪个模块？
2. 系统是在哪一层决定用 `C++` 还是 `Python` runner？
3. 测试点数据是在哪一层加载的？
4. 最终 verdict 是谁决定的？
5. 结果最后是怎么回给客户端的？

如果这 5 个问题你能说出大概意思，那说明你已经真的顺着主线理解项目了。

---

## 二十一、最后一句鼓励

你现在不需要记住所有函数名。

你只需要先记住这条主线：

**收请求 -> 翻译请求 -> 调度执行 -> 跑测试点 -> 汇总结果 -> 回包**

这条线一旦记住，后面你再看任何文件，都会更容易。
