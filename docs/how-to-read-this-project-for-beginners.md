# 给代码小白的阅读指南

这份文档不是教你“一次看懂所有代码”。

它的目标只有一个：

**让你第一次读这个项目时，知道应该先看哪里、后看哪里、每一步到底在看什么。**

如果你现在几乎还是代码小白，也没关系。你不需要先学会所有 C++ 语法，先学会“怎么看一个项目”更重要。

---

## 一、先用一句话理解这个项目

这个项目本质上是一个：

**“接收用户提交的代码 -> 编译/运行 -> 对照标准答案 -> 返回评测结果”**

你可以把它想成一个老师批改作业的系统：

- 学生交一份代码
- 系统拿这份代码去做题
- 系统看输出对不对
- 系统告诉你：通过了，还是错了，还是超时了，还是编译失败了

这个项目现在重点支持两种语言：

- `C++`
- `Python`

---

## 二、先不要急着看细节代码

第一次读项目，最容易犯的错误是：

**一上来就钻进某个 `.cpp` 文件，一行一行死读。**

这样很容易看 10 分钟就迷路。

正确做法是先建立“地图”。

你先记住这 5 层：

1. `protocol`：协议层，负责把网络消息变成程序能懂的数据
2. `service`：服务层，负责串起整次评测流程
3. `runner`：执行层，负责具体跑 `C++` 或 `Python`
4. `judge`：判题层，负责汇总结果
5. `store`：存储层，负责保存当前评测状态

你可以把它理解成一条流水线：

**收到请求 -> 解释请求 -> 执行代码 -> 汇总结果 -> 保存结果 -> 回给客户端**

---

## 三、最适合你的阅读顺序

下面这条顺序，是我专门按“代码小白第一次阅读”设计的。

### 第 1 步：先看项目说明，不看代码

先看这两个文件：

- `README.md`
- `docs/architecture.md`

你现在只需要回答这 3 个问题：

1. 这个项目是干什么的？
2. 它现在支持什么语言？
3. 它大概分成哪几层？

如果这 3 个问题你能说出来，说明你已经比“直接冲进代码里”强很多了。

---

### 第 2 步：看程序从哪里启动

看：

- `main.cpp`

你现在不要管每一行。

你只看这几件事：

- 程序从哪里开始运行
- 它有没有读取配置文件
- 它有没有启动 TCP server
- 它把连接事件交给了谁处理

读完后，你心里要有一个很简单的印象：

**“main 负责启动整个系统，不负责具体判题。”**

这很重要。

---

### 第 3 步：看网络请求是怎么进来的

看：

- `src/TcpServer.cpp`
- `src/client_sockets.cpp`
- `include/client_sockets.h`

你现在只看“大动作”，不要深究底层 socket 细节。

你只需要知道：

- `TcpServer` 在监听端口
- `ClientSockets` 在处理每个客户端连接
- 它会先收消息
- 再把消息交给新判题链路
- 最后把结果发回去

如果看到 `select`、`fd_set`、`socket` 这些词有点晕，很正常。

你此时不用精通它们，只要知道：

**这部分代码是在做“网络收发”和“连接管理”。**

---

### 第 4 步：看“请求”是怎么被翻译成程序内部数据的

看：

- `include/protocol/JudgeProtocol.h`
- `src/protocol/JudgeProtocol.cpp`

这里是非常适合新手读的一层，因为它的职责很单纯。

你重点看：

- 输入是什么：JSON 请求
- 输出是什么：`SubmissionRequest`
- 结果怎么编码成 JSON 返回

读这一层时，你要形成一个观念：

**协议层不负责判题，它只负责“翻译”。**

就像翻译官，不负责决定答案对不对。

---

### 第 5 步：看整个评测流程的“大脑”

看：

- `include/service/SubmissionService.h`
- `src/service/SubmissionService.cpp`

这是你阅读的重点文件之一。

因为它最像“总调度员”。

你重点找这几个阶段：

- `PREPARING`
- `COMPILING`
- `RUNNING`
- `FINISHED`
- `FAILED`

你可以边看边问自己：

1. 它什么时候创建一次提交？
2. 它什么时候选择 runner？
3. 它什么时候编译？
4. 它什么时候逐个测试点运行？
5. 它什么时候汇总最终结果？

如果你能顺着这条线看懂，整个项目就已经看懂一半了。

---

### 第 6 步：看“不同语言怎么分别执行”

先看接口：

- `include/runner/ILanguageRunner.h`
- `include/runner/RunnerFactory.h`
- `src/runner/RunnerFactory.cpp`

再看两个具体实现：

- `src/runner/CppRunner.cpp`
- `src/runner/PythonRunner.cpp`

这里你要建立一个关键概念：

**“同样是判题流程，不同语言只是换了不同 runner。”**

也就是说：

- 流程基本一样
- 但 `C++` 要编译
- `Python` 主要做语法检查 + 运行

你第一次看时，先不要追所有系统调用。

你只要看懂：

- `prepare()` 是干嘛的
- `compile()` 是干嘛的
- `runCase()` 是干嘛的

---

### 第 7 步：看最后怎么得出 verdict

看：

- `include/judge/JudgeCore.h`
- `src/judge/JudgeCore.cpp`

这里的职责也很简单：

**把多个测试点结果汇总成一个总结果。**

比如：

- 全部对 -> `AC`
- 编译失败 -> `CE`
- 超时 -> `TLE`
- 运行错误 -> `RE`

这层很像“批改总结”。

---

### 第 8 步：看结果保存在哪里

看：

- `include/store/ResultStore.h`
- `src/store/ResultStore.cpp`

你现在只看这几个点：

- 提交结果是怎么保存的
- 状态能不能乱跳
- 为什么说状态只能往前走

这是个“仓库管理员”角色。

它不跑代码，只保存当前进度和结果。

---

### 第 9 步：最后再看测试

很多新手会忽略测试，但其实测试是最好的老师。

推荐按这个顺序看：

- `test/test_protocol_codec.cpp`
- `test/test_submission_service.cpp`
- `test/test_cpp_runner.cpp`
- `test/test_python_runner.cpp`
- `test/test_integration_tcp_cpp_python.cpp`

为什么？

因为测试文件通常比正式代码更直接。

它会明确告诉你：

- 输入是什么
- 期望输出是什么
- 作者希望这个模块怎么工作

如果你读实现代码读晕了，回头看测试，往往会突然清楚很多。

---

## 四、你第一次阅读时，可以先跳过什么

为了不把自己吓住，第一次你可以先跳过这些：

- 太底层的 socket 细节
- `Buffer` 内部实现细节
- 所有 `mutex` / 线程同步细节
- 旧链路里 `testBox` / `resultContainer` 的所有历史包袱
- 特别长的工具函数实现

你先抓主线，不要一开始就想“每一行都看懂”。

第一次阅读最重要的是：

**知道每个模块是干嘛的。**

不是每个细节都吃透。

---

## 五、每次读代码时，用这 4 个问题逼自己思考

无论你在看哪个文件，都先问：

1. 这个文件负责什么？
2. 它接收什么输入？
3. 它输出什么结果？
4. 它依赖谁？

只要这 4 个问题能答出来，你就已经在“真正理解代码”，而不是机械看字母了。

---

## 六、如果看着看着卡住了，怎么办

这是最实用的一段。

### 情况 1：看不懂某个函数在干嘛

做法：

- 不要死磕整段实现
- 先看这个函数的名字
- 再看它的参数和返回值
- 再看是谁调用了它

很多时候，调用关系比函数内部细节更容易懂。

### 情况 2：看不懂某个类为什么存在

做法：

- 看它被谁创建
- 看它最重要的 2~3 个方法
- 看测试里怎么使用它

### 情况 3：术语太多，看着头大

做法：

- 先把术语翻成普通话

比如：

- `protocol` = 协议翻译层
- `submission` = 一次提交
- `runner` = 代码执行器
- `verdict` = 判题结论
- `store` = 结果仓库

### 情况 4：一个文件太长，读不下去

做法：

- 先只找“主流程函数”
- 忽略 helper 小函数
- 只画出调用顺序

你甚至可以先在纸上写成：

`收到请求 -> 解析 -> 提交 -> 编译 -> 跑用例 -> 汇总 -> 回包`

只要主线没丢，就不会迷路。

## 七、推荐你的第一次阅读用时

如果你是代码小白，我建议你这样分配：

### 第一天

- `README.md`
- `docs/architecture.md`
- `main.cpp`
- `src/client_sockets.cpp`

目标：知道“请求是怎么进来的”。

### 第二天

- `src/protocol/JudgeProtocol.cpp`
- `src/service/SubmissionService.cpp`

目标：知道“请求进来以后，整个判题流程怎么走”。

### 第三天

- `src/runner/CppRunner.cpp`
- `src/runner/PythonRunner.cpp`
- `src/judge/JudgeCore.cpp`
- `src/store/ResultStore.cpp`

目标：知道“代码怎么执行、结果怎么汇总、状态怎么保存”。

### 第四天

- 重点测试文件

目标：通过测试反过来验证自己的理解。

## 九、小白词典

### `Submission`

一次代码提交。

### `Protocol`

负责把网络消息翻译成程序能用的数据结构。

### `Runner`

真正去跑代码的人。

### `Judge`

根据测试点结果，得出最后结论的人。

### `Verdict`

最后判题结果，比如：

- `AC`：通过
- `WA`：答案错
- `CE`：编译错误
- `RE`：运行错误
- `TLE`：超时

### `Store`

保存当前评测状态和结果的地方。
