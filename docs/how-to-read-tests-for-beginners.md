# 给代码小白的测试文件怎么读

很多代码小白看到测试文件时，会本能地觉得：

“这个应该是高级内容，我先不看。”

其实正好相反。

**测试文件往往比正式实现更适合新手读。**

因为测试会直接告诉你：

- 输入是什么
- 预期输出是什么
- 作者认为“正确行为”应该是什么

所以如果实现代码太绕，你反而应该先看测试。

---

## 一、先记住：测试文件像“标准答案说明书”

你可以把测试文件想成这样：

- 正式代码 = 真正做事的人
- 测试代码 = 检查这个人有没有按要求做事的人

所以测试不是多余的。

测试其实是在告诉你：

**这个模块到底应该怎么工作。**

---

## 二、你应该先看哪些测试

如果你是第一次读这个项目，我建议按这个顺序：

1. `test/test_protocol_codec.cpp`
2. `test/test_submission_service.cpp`
3. `test/test_cpp_runner.cpp`
4. `test/test_python_runner.cpp`
5. `test/test_judge_core.cpp`
6. `test/test_result_store.cpp`
7. `test/test_integration_tcp_cpp_python.cpp`

这个顺序的逻辑是：

- 先看简单的协议
- 再看主流程
- 再看语言执行
- 再看判题和状态
- 最后看端到端集成

---

## 三、怎么读一份测试文件

你不要一行一行机械读。

你应该按这个顺序读：

### 第 1 步：先看测试函数名字

比如：

- `test_decode_request_uses_current_json_shape`
- `test_compile_failure_finishes_with_ce`
- `test_tcp_cpp_submission_also_flows_through_service_and_returns_ac`

这些名字其实已经在告诉你作者想验证什么。

### 第 2 步：看输入是什么

比如：

- 传了什么 JSON
- 构造了什么 `SubmissionRequest`
- 提交了什么 C++/Python 代码

### 第 3 步：看断言在检查什么

重点看：

- `assert(...)`

因为断言就是测试真正关心的东西。

### 第 4 步：最后再去对照实现代码

当你知道测试“希望发生什么”之后，再去看实现代码，你会轻松很多。

---

## 四、逐个解释最值得读的测试

## 1. `test/test_protocol_codec.cpp`

### 它在测什么

它在测：

- JSON 请求能不能正确解析
- 结果能不能正确编码回 JSON
- 错误包格式对不对

### 你第一次重点看什么

你重点看这些测试：

- 请求里有 `uuid/pid/lang/code`
- 长 `pid` 会不会被截断
- 错误类型不对时会不会安全失败
- 返回 JSON 里有哪些字段

### 你读完应该知道什么

**协议层希望输入输出都非常稳定。**

也就是说，只要协议层测试过了，你就知道客户端和服务端之间“说话的格式”是明确的。

---

## 2. `test/test_submission_service.cpp`

### 它在测什么

它在测整个评测主流程。

### 你第一次重点看什么

你重点看这些测试函数：

- 成功提交后是否能拿到结果
- 成功路径最后是不是 `FINISHED + AC`
- 编译失败是不是 `CE`
- 不支持语言是不是 `SYSTEM_ERROR`
- 查不存在的提交是不是 `false`

### 为什么这个测试很重要

因为它最接近“整个系统应该怎么工作”。

如果你只读一个测试文件，我最推荐你读这个。

### 你读完应该知道什么

**`SubmissionService` 才是整个评测流程的总调度员。**

---

## 3. `test/test_cpp_runner.cpp`

### 它在测什么

它在测 `CppRunner`：

- 能不能准备源码文件
- 能不能编译成功
- 编译失败时会不会返回正确结果
- 运行单个测试点时会不会给出正确 verdict

### 你第一次重点看什么

你主要看：

- 小的 C++ 程序是怎么被喂给 runner 的
- 成功路径怎么断言
- 编译失败怎么断言
- 运行错误怎么断言

### 你读完应该知道什么

**C++ runner 的责任很具体：写文件、编译、执行。**

---

## 4. `test/test_python_runner.cpp`

### 它在测什么

它在测 `PythonRunner`：

- 语法正确时能否通过检查
- 语法错误时是否给 `CE`
- 正常输出时是否 `AC`
- 运行异常时是否 `RE`
- 死循环时是否 `TLE`

### 你第一次重点看什么

你重点看：

- Python 是怎么统一放进 `prepare/compile/runCase` 这套流程里的
- 为什么语法检查会算“compile”阶段

### 你读完应该知道什么

**Python 虽然不是编译型语言，但为了统一系统流程，也被放进了 compile-check 模式。**

---

## 5. `test/test_judge_core.cpp`

### 它在测什么

它在测多个 case 结果怎么汇总。

### 你第一次重点看什么

你看 verdict 优先级就行：

- 全 `AC` 怎么办
- 有 `CE` 怎么办
- 有 `TLE` 怎么办
- 有 `RE` 怎么办

### 你读完应该知道什么

**最后的总 verdict 不是拍脑袋决定的，而是有规则的。**

---

## 6. `test/test_result_store.cpp`

### 它在测什么

它在测存储层：

- 状态能不能正常前进
- 不合法状态跳转会不会被拒绝
- 查不到 submission 时会不会返回 `false`
- 并发读写会不会坏掉

### 你第一次重点看什么

你重点看状态机就行。

### 你读完应该知道什么

**`ResultStore` 不是随便改状态的，它有严格顺序。**

---

## 7. `test/test_integration_tcp_cpp_python.cpp`

### 它在测什么

它在做端到端测试。

也就是从“像客户端一样发请求”，一直测到“收到最终 JSON 响应”。

### 你第一次重点看什么

你重点看：

- 请求是怎么发进去的
- 返回 JSON 有哪些字段
- `Python` 和 `C++` 是否都能走通

### 为什么它很重要

因为它最像真实用户在用系统。

### 你读完应该知道什么

**新链路不是只在单元测试里成立，而是真的能从 TCP 入口一路跑通。**

---

## 五、`test/CMakeLists.txt` 为什么也值得看

文件：

- `test/CMakeLists.txt`

它在告诉你：

- 哪些 `test*.cpp` 会被自动当作测试
- 哪些文件不会被默认加入 CTest

这对你理解“哪些是正式自动化测试，哪些只是手工调试程序”很有帮助。

---

## 六、为什么测试有时比实现更适合你

因为实现代码经常有这些问题：

- 历史兼容逻辑
- 系统调用
- 线程
- 文件系统操作

而测试通常更直接：

- 给输入
- 看输出
- 做断言

所以对代码小白来说，测试文件常常更像“中文说明书”。

---

## 七、如果你看测试也看晕了，怎么办

这里有一个很实用的办法。

每读一个测试，只做这三件事：

1. 写下它的测试目标
2. 写下它的输入
3. 写下它最后断言了什么

比如你可以写成：

```text
这个测试在检查：编译失败时 SubmissionService 会返回 CE
输入：一段坏掉的 Python 代码
断言：status == FINISHED, verdict == CE
```

只要你能写出这三句话，这个测试你就已经读懂了大半。

---

## 八、最后给你的建议

如果你今天时间不多，我建议你只看这 3 个测试文件：

1. `test/test_protocol_codec.cpp`
2. `test/test_submission_service.cpp`
3. `test/test_integration_tcp_cpp_python.cpp`

为什么是这 3 个？

因为它们刚好对应：

- 请求怎么进来
- 主流程怎么跑
- 端到端怎么完成

只看完这 3 个，你就已经会比只看实现代码清楚得多。
