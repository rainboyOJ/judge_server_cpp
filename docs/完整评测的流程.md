# 完整评测的流程

这篇文档记录 `judge_server` 从收到一次 `submit` 请求，到返回最终评测结果的完整路径。读完后应该能回答三个问题：

1. 一次提交在代码里经过哪些模块。
2. 每个模块只负责什么，不负责什么。
3. 为什么当前实现是“网络线程快速入队，后台 worker 慢慢评测”。

## 1. 总体视角

`judge_server` 的核心思想是把一次评测拆成两条线：

```text
客户端 TCP 请求
  -> 网络层解析协议
  -> SubmissionService 创建 submission 并放入内部队列
  -> 立即返回 submission_ack

后台 worker 线程
  -> 从 SubmissionService 内部队列取任务
  -> prepare/compile/run case
  -> 写入 ResultStore
  -> 通过网络层回推 submission_finished
```

也就是说，网络层不会直接编译和运行用户代码。网络层只负责：

- 接收 TCP 消息。
- 解码 JSON。
- 调用 `SubmissionService::submitAsync()`。
- 返回 ack、查询结果、回推异步结果。

真正耗时的评测工作在 `JudgeWorkerPool` 的后台线程里执行。

## 2. 启动时创建了哪些对象

入口在 `src/main.cpp`。

启动时主要创建这些对象：

```cpp
ResultStore result_store;
RunnerFactory runner_factory;
SubmissionVerdictReducer judge_verdict_reducer;
SubmissionService submission_service(result_store, runner_factory, judge_verdict_reducer);

ClientSockets client_sockets(..., submission_service, ...);
JudgeWorkerPool judge_worker_pool(..., submission_service, &client_sockets);
TcpServer server(...);
```

它们的关系是：

| 对象 | 职责 |
| --- | --- |
| `TcpServer` | 监听端口，accept 连接，用 `select` 分发读写事件 |
| `ClientSockets` | 管理所有客户端连接，负责协议接入、回包和异步结果投递 |
| `SubmissionService` | 提交服务入口，内部持有待评测队列，负责完整评测编排 |
| `JudgeWorkerPool` | 后台 worker 线程池，从 `SubmissionService` 取任务并执行评测 |
| `ResultStore` | 保存每个 submission 的最新状态快照 |
| `RunnerFactory` | 根据语言创建 C++/Python runner |
| `SubmissionVerdictReducer` | 把多个测试点结果汇总成最终 verdict |

需要注意：`SubmissionQueue` 现在是 `SubmissionService` 的内部实现细节。`main.cpp`、网络层、worker pool 都不直接持有队列对象。

## 3. 请求协议进入服务端

客户端发送的是：

```text
4 字节 big-endian 长度 + JSON body
```

`JSON body` 的提交请求形状大致是：

```json
{
  "type": "submit",
  "uuid": 1,
  "pid": "1000",
  "lang": 0,
  "code": "#include <iostream>\nint main(){...}"
}
```

协议解析在 `src/protocol/JudgeProtocol.cpp`：

- `decodeRequest()` 解析 `type=submit`。
- `decodeQueryRequest()` 解析 `type=query_result`。
- `encodeSubmissionAck()` 生成入队确认。
- `encodeSubmissionFinished()` 生成最终结果。
- `encodeResult()` 根据当前状态生成查询返回。

`lang` 当前约定：

| 值 | 语言 |
| --- | --- |
| `0` | C++ |
| `1` | C，枚举保留，但当前 `RunnerFactory` 未创建 C runner |
| `2` | Python |

## 4. 网络层如何处理一次 submit

网络事件入口在 `ClientSockets::deal_events()`。

一次读事件大致经过：

```text
ClientSockets::deal_events()
  -> handle_read_event()
  -> ConnectionSlot::read_message()
  -> handle_complete_message()
  -> JudgeProtocol::decodeRequest()
  -> handle_submit_message()
  -> SubmissionRequestHandler::handleSubmit()
```

`ConnectionSlot::read_message()` 负责从 socket 中凑齐一个完整的 length-prefixed 消息。只有完整 JSON body 到达后，才进入 `handle_complete_message()`。

`handle_complete_message()` 会先尝试把消息当成 `submit` 解码。如果不是 `submit`，再尝试当成 `query_result`。两者都不是时返回 `bad request`。

## 5. 为什么需要 reply_channel

`submit` 是异步的。客户端连接还活着时，worker 评测完成后需要知道把结果发回哪个连接。

所以网络层会为当前连接生成一个 `ReplyChannel`：

```text
slot_id + session_id
```

含义是：

- `slot_id`：连接在 `ConnectionRegistry` 里的槽位。
- `session_id`：这个槽位当前连接的代号。

这样可以避免一个旧任务完成后，把结果误发给后来复用同一个槽位的新连接。

## 6. submitAsync 做了什么

`SubmissionRequestHandler::handleSubmit()` 会调用：

```cpp
submission_service_.submitAsync(request, reply_channel_id);
```

`SubmissionService::submitAsync()` 做三件事：

1. 调用 `createSubmission()`，在 `ResultStore` 中创建一份 submission，初始状态是 `QUEUED/PENDING`。
2. 构造 `SubmissionTask{submission_id, request, reply_channel_id}`。
3. 把任务写入 `SubmissionService` 内部队列。

入队成功后，网络层立即返回：

```json
{
  "type": "submission_ack",
  "submission_id": 1,
  "status": "QUEUED",
  "verdict": "PENDING",
  "message": "queued",
  "case_results": []
}
```

这个 ack 只表示服务端已经接收并排队，不表示已经编译或运行。

## 7. AckBarrier 解决什么问题

有一种竞态需要处理：

```text
网络线程刚把任务入队
后台 worker 立刻取到任务并评测完成
最终结果可能比 submission_ack 更早准备好
```

如果直接发送，客户端可能先收到 `submission_finished`，后收到 `submission_ack`，这会让协议顺序很难理解。

`AckBarrier` 的作用是保证：

```text
同一个 reply_channel 上，submission_ack 一定先于异步 started/finished 消息发送
```

流程是：

1. `handleSubmit()` 先 `mark_waiting(reply_channel_id)`。
2. 调用 `submitAsync()` 入队。
3. 生成 `submission_ack`。
4. `release(reply_channel_id)`，释放期间被 defer 的异步消息。

如果 worker 在 ack 发送前已经产生消息，`ClientSockets::queue_protocol_response_for_channel()` 会通过 `AckBarrier::try_defer()` 暂存它。

## 8. worker 如何取任务

worker 线程在 `src/dispatch/JudgeWorkerPool.cpp`。

核心循环是：

```text
while (service_.waitTask(task)) {
  notifier->onSubmissionStarted(task)
  service_.processSubmission(task.submission_id, task.request)
  notifier->onSubmissionFinished(task)
}
```

这里的 `service_` 就是 `SubmissionService`。

也就是说，`JudgeWorkerPool` 不知道队列的具体实现，只知道：

- `waitTask()` 能阻塞等待一个待评测任务。
- `processSubmission()` 能执行完整评测。
- `shutdownQueue()` 能让 worker 在服务停止时退出。

## 9. processSubmission 的状态机

真正的评测编排在 `SubmissionService::processSubmission()`。

一次正常评测的状态变化是：

```text
QUEUED
  -> PREPARING
  -> COMPILING
  -> RUNNING
  -> FINISHED
```

异常时可能进入：

```text
FAILED / SYSTEM_ERROR
```

每次状态变化都会写入 `ResultStore`，所以客户端可以用 `query_result` 查到当前最新状态。

## 10. PREPARING 阶段

`PREPARING` 表示服务端准备运行环境。

在这一阶段，`SubmissionService` 会：

1. 根据 `request.language` 调用 `RunnerFactory::createRunner()`。
2. 如果语言不支持，写入 `FAILED/SYSTEM_ERROR`。
3. 把 `request.uuid` 改成服务端生成的 `submission_id`。
4. 调用 runner 的 `prepare()`。

把 `uuid` 改成 `submission_id` 的原因是：服务端需要用自己的唯一编号创建工作目录，避免客户端传入重复 uuid 时互相覆盖编译产物。

## 11. COMPILING 阶段

`COMPILING` 表示 runner 正在处理编译动作。

对 C++：

- `prepare()` 通常负责写入源码文件。
- `compile()` 调用编译器生成可执行文件。
- 编译失败会得到 `FINISHED/CE`。

对 Python：

- Python 是解释型语言，但为了保持状态机统一，也会经过 `compile()` 阶段。
- 这里可以理解成语法检查或准备运行文件。

如果编译失败，评测不会进入测试点运行阶段。

## 12. RUNNING 阶段

编译成功后进入 `RUNNING`。

`SubmissionService` 会根据 `pid` 加载测试数据：

```text
testData/<pid>/data/
```

每组测试点包含输入文件和标准输出文件。当前测试点限制在 `load_cases_for_problem()` 中固定为：

```text
cpu_time_limit_ms = 1000
real_time_limit_ms = 1000
memory_limit_kb = 1024 * 1024
```

然后依次执行每个 case：

```text
runner->runCase(execution_request, case_input)
  -> run_executable_case(...)
  -> run_sjudger(config)
  -> compare_case_output(...)
```

每跑完一个 case，当前 `case_results` 都会写入 `ResultStore`。因此如果客户端在运行中查询，理论上可以看到已完成的测试点结果。

## 13. sjudge 在哪里发挥作用

runner 不直接 `fork/exec` 用户程序，而是通过 `run_sjudger(config)` 进入 `sjudge`。

当前 `run_executable_case()` 会把这些信息填入 `judge_config`：

- 可执行文件路径。
- 工作目录。
- stdin 输入文件。
- stdout 输出文件。
- stderr 文件。
- CPU 时间限制。
- 真实时间限制。
- 内存限制。
- 低权限运行的 `uid/gid`。
- 是否清理 supplementary groups。
- 是否启用 seccomp。
- 是否禁止网络相关 syscall。
- 是否禁止 fork/clone 等创建进程行为。
- cgroup 路径和资源限制参数。

`sjudge` 的职责是安全地运行用户程序：

```text
父进程
  -> fork 子进程
  -> 监控真实时间、CPU 时间、内存、退出状态、信号

子进程
  -> chdir 到工作目录
  -> 重定向 stdin/stdout/stderr
  -> 设置 rlimit
  -> 可选加入 cgroup
  -> 可选 setgid/setuid 降权
  -> 可选加载 seccomp 规则
  -> execve 用户程序
```

`sjudge` 返回的是运行层面的结果，例如：

- 正常退出。
- CPU 超时。
- 真实时间超时。
- 内存超限。
- 运行时错误。
- sandbox 自己出错。

## 14. 为什么运行成功不等于 AC

用户程序执行成功，只说明它没有超时、没有超内存、没有崩溃。

还必须比较输出：

```text
用户输出 vs 标准输出
```

当前比较逻辑在 `compare_case_output()`：

1. 如果存在 `/judge/checker/fcmp2`，优先调用外部 checker。
2. 如果没有 checker，退化为内置按行比较。

内置比较会忽略：

- 行尾空格和 tab。
- 文件末尾多余空行。

所以一个 case 的判定顺序是：

```text
sjudge 运行结果不是成功
  -> TLE/MLE/RE/SYSTEM_ERROR

sjudge 运行成功
  -> 再比较输出
  -> AC 或 WA
```

## 15. 最终 verdict 如何汇总

所有 case 执行结束后，`SubmissionService` 调用：

```cpp
judge_verdict_reducer_.summarize(running_result.case_results)
```

`SubmissionVerdictReducer` 负责把多个测试点结果汇总成 submission 的最终 verdict。

例如：

- 全部 AC，最终 AC。
- 任意一个 WA，最终 WA。
- 任意一个 TLE，最终 TLE。
- 任意一个 MLE，最终 MLE。
- runner 或系统异常，最终 SYSTEM_ERROR。

最终结果会写入 `ResultStore`，状态变成：

```text
FINISHED
```

## 16. 结果如何回推给客户端

worker 执行完成后会调用：

```text
notifier->onSubmissionFinished(task)
```

这里的 notifier 实际上是 `ClientSockets`。

回推路径是：

```text
JudgeWorkerPool
  -> ClientSockets::onSubmissionFinished()
  -> SubmissionEventResponder::onSubmissionFinished()
  -> SubmissionService::query()
  -> JudgeProtocol::encodeSubmissionFinished()
  -> queue_protocol_response_for_channel()
  -> wake_select_loop()
  -> TcpServer 被 eventfd 唤醒
  -> handle_write_event()
  -> socket send()
```

`SubmissionEventResponder` 不保存结果。它总是回查 `SubmissionService::query()`，拿 `ResultStore` 里的最新快照编码成协议消息。

## 17. 如果客户端断开了怎么办

如果客户端断开：

- `ConnectionRegistry` 会释放连接槽位。
- 后台 worker 仍然会继续完成评测。
- 最终结果仍然写在 `ResultStore` 中。
- 异步回推会因为 `reply_channel` 无效或 `session_id` 过期被丢弃。

也就是说，断开连接不会取消评测。

客户端之后如果知道 `submission_id`，可以重新连接并发送：

```json
{
  "type": "query_result",
  "submission_id": 1
}
```

服务端会从 `ResultStore` 查询并返回当前状态。

## 18. 一次完整流程时序图

```text
Client
  |
  | 4-byte length + {"type":"submit",...}
  v
TcpServer / ClientSockets
  |
  | read_message + decodeRequest
  v
SubmissionRequestHandler
  |
  | AckBarrier::mark_waiting
  | SubmissionService::submitAsync
  v
SubmissionService
  |
  | ResultStore::createSubmission -> QUEUED/PENDING
  | internal queue push SubmissionTask
  v
SubmissionRequestHandler
  |
  | encodeSubmissionAck
  | AckBarrier::release
  v
ClientSockets
  |
  | send submission_ack
  v
Client

JudgeWorkerPool worker
  |
  | SubmissionService::waitTask
  v
SubmissionService::processSubmission
  |
  | PREPARING -> RunnerFactory -> runner.prepare
  | COMPILING -> runner.compile
  | RUNNING -> load cases -> runner.runCase
  | per case -> run_sjudger -> compare output -> ResultStore update
  | FINISHED -> SubmissionVerdictReducer::summarize -> ResultStore update
  v
JudgeWorkerPool worker
  |
  | notifier.onSubmissionFinished
  v
ClientSockets
  |
  | query latest result
  | encodeSubmissionFinished
  | queue response + wake select loop
  v
Client
```

## 19. 当前实现的边界

当前实现已经符合一个小型 judge server 的主流程，但还不是完整生产级 OJ 后端。需要知道这些边界：

- 队列是内存队列，没有持久化；进程重启后排队任务会丢失。
- `ResultStore` 是内存存储，不是数据库；进程重启后结果会丢失。
- 没有取消 submission 的协议。
- 没有重试机制。
- 没有优先级队列。
- 测试点限制目前在 `SubmissionService::load_cases_for_problem()` 中固定，没有从题目配置读取。
- 当前支持 C++ 和 Python；C 枚举存在，但 runner 尚未实现。
- 异步回推目前主要发送 started/finished，未对每个 case 单独主动推送更新。
- 低权限用户、seccomp、cgroup 等安全能力依赖运行环境配置；配置错误时隔离强度会下降。

## 20. 阅读源码建议

按这个顺序看源码最容易串起来：

1. `src/main.cpp`：看对象如何组装。
2. `src/network/TcpServer.cpp`：看连接事件循环。
3. `src/network/ClientSockets.cpp`：看 submit/query 如何分发。
4. `src/network/SubmissionRequestHandler.cpp`：看 submit 如何入队并返回 ack。
5. `src/pipeline/SubmissionService.cpp`：看完整评测状态机。
6. `src/dispatch/JudgeWorkerPool.cpp`：看后台 worker 如何取任务。
7. `src/runner/CppRunner.cpp` 和 `src/runner/PythonRunner.cpp`：看语言 runner 如何 prepare/compile/run。
8. `src/runner/RunnerExecutionSupport.cpp`：看如何调用 `sjudge` 和比较输出。
9. `sjudge/`：看 sandbox 如何 fork、降权、限制资源和等待结果。

如果只想抓住主线，重点读 `ClientSockets -> SubmissionRequestHandler -> SubmissionService -> JudgeWorkerPool -> RunnerExecutionSupport` 这条路径。
